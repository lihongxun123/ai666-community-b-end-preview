B端 V0.1 走查规则

本原型验证 Ant Design 后台信息架构、24 个二级页面、21 个正式 C端页面承接、广告位配置闭环、首页登录后弹窗配置闭环、审核处理闭环、活动任务积分闭环、用户积分权限日志闭环、消息商城数据复盘闭环、受控批处理、抽屉详情、异常态、危险操作确认和敏感信息脱敏。

正式页清理规则
  • 页面主体不展示业务契约、验收合同、来源说明和研发口径。
  • 规则卡、流程卡、保护边界、复杂度边界和“不可做”说明统一收纳到走查。
  • 正式页面只保留后台用户需要操作的标题、指标、筛选、表格、批量动作、抽屉、确认弹窗和状态。
  • 全局顶部不展示面包屑、角色标签、用户 ID 或退出文字，只保留当前页标题和登录用户头像入口。
  • 验证用 `data-*` 属性保留在 DOM 中，但不作为正式页面可见元素展示。
规则类信息收纳范围
  • 配置页面字段必须先映射到 C端真实页面、入口、任务、活动、消息、商城或用户状态；没有 C端来源的字段不得为了“后台完整”单独捏造。
  • 新增和编辑优先使用同一个表单弹窗；编辑从列表行填充，新建只保留系统生成 ID、默认状态等必要默认值。
  • 业务对象 ID 由系统自动生成并只读展示，不允许运营手填或修改。
  • 二元展示状态使用开关，多状态只使用受控胶囊或选项，不在列表里暴露任意状态文案。
  • 列表只保留识别、扫描和进入编辑所需字段；调序、目标替换、启停、素材补充等配置动作收进编辑弹窗。
  • 规则、触发条件、频控、灰度、异常阻断和日志口径进入走查规则，不作为正式页面主内容。
  • 一级导航固定为 6 个。
  • 二级页面固定为 24 个，页面内必须有业务契约。
  • 广告位配置只展示本期确认属于运营投放的广告位：首页 Banner、首页金刚区、首页爆款首卡、全站顶部活动条、闪念首屏广告、闪念右侧广告和模型页 Hero Banner。
  • 首页信息流广告本期不做，不在正式列表和页签出现；后续需要单独补齐信息流插卡位置、频控、素材规格和排序规则。
  • 首页登录后弹窗归属弹窗配置；公告入口归属后续公告配置；AIGC 顶部背景视频归属页面素材配置；积分中心和邀请固定入口不进入广告位配置。
  • 首页 Banner 和模型页 Hero 按多条单张广告组织，轮播关系由同一页签下的排序字段表达，不单独创建组合配置行。
  • 爆款首卡广告、全站顶部活动条、闪念首屏广告和闪念右侧广告按单广告位组织，列表中各出现一个配置对象。
  • 广告位配置对象 ID 由系统自动生成，不作为运营可见编辑项；类型、广告标识、整卡目标、有效期和负责人不进入正式编辑弹窗。
  • 广告位编辑弹窗的页面位置选项必须和外部广告位页签保持一致；选择首页金刚区时展示 Icon 字段并写入首页金刚区表格；背景图片使用 Ant Design 上传组件形态，排序和状态固定放在最后一行。
  • 全站顶部活动条不维护背景图片；切换到该页签时隐藏背景图片列，编辑弹窗选择该点位时隐藏背景图片上传字段。
  • 广告位配置必须覆盖点位选择、目标绑定、展示字段校验、启停排序和日志记录。
  • 广告位新增和编辑必须使用同一个广告位配置弹窗；列表只保留标题、描述、背景图片、状态、排序和操作，编辑弹窗只保留页面位置、标题、描述、背景图片、条件 Icon、是否启用和排序。
  • 活动配置统一承接活动中心内容；新手任务、七日成长任务、邀请有礼均按活动管理，任务配置不再作为独立运营配置板块。
  • 活动配置的基础信息只维护活动标题、活动时间、活动封面图和活动详情；列表只保留标题、活动时间、封面图和编辑入口。
  • 活动时间使用 Ant Design 时间段选择器形态；活动封面使用 Ant Design 上传组件形态；活动详情使用 Markdown 编辑器承接概述、参与路径、奖励说明和规则描述。
  • 积分奖励按 5 类活动规则进入对应活动的编辑弹窗：投稿型、内容共创型、新手任务型、连续成长型和邀请阶梯型；每类规则直接维护触发条件、积分数值、上限、确认方式和异常处理，不在列表页设置独立规则面板。
  • AI 生图活动奖励与七日成长 D5 奖励必须分开记录，不能合并成同一条积分规则。
  • Prompt 纯文字和带案例按奖励档位处理，不默认叠加为 60 积分；邀请有礼以单个好友最高 90 积分为规则口径，今日和本月数据只做运营展示。
  • 活动配置只允许编辑既有活动，不提供新建活动入口；新增活动涉及 C端页面、活动逻辑和奖励规则同步，应作为独立研发需求处理。
  • 内容运营页面新增、编辑和处理必须使用同一个内容运营处理弹窗；字段来自内容列表、审核工作台、闪念管理、模板配置和教程管理，投稿来源内容统一进入内容列表，首页爆款状态由内容列表承接，ID 自动生成且只读，风险动作、用户可读原因和内部备注统一收进弹窗和日志。
  • 用户管理页面处理必须使用同一个用户治理弹窗；字段来自用户列表、用户详情记录、邀请记录、积分记录和黑白名单，敏感信息保持脱敏，处理原因和权限/积分变更必须留痕。
  • 登录后弹窗主体只允许配置弹窗标题和是否弹出，不展示 ID、触发场景、子项数、默认子项、负责人或列表操作。
  • 弹窗子项必须覆盖标题、描述、活动大图、按钮文案和跳转链接；列表标题取 C端标题逗号前内容，描述列展示奖励积分，状态胶囊只允许上架或下架，排序字段仅展示，调整排序合并到编辑弹窗。
  • 弹窗子项新增和编辑必须使用同一个表单弹窗；编辑从列表行填充内容，新增除自动生成 ID 和默认下架外为空表单，保存入口一致。
  • 弹窗子项 ID 由系统自动生成并隐藏在表单数据中，不作为运营可见编辑项；活动大图使用 Ant Design 上传组件形态，排序和状态固定放在表单最后一行。
  • 后台表单不放用户侧预览组件；用户侧组件形态、信息层级和展示校验统一收纳到走查规则，不进入正式配置弹窗。
  • 弹窗子项默认值必须同步 C端任务强引导 taskGuideTasks，顺序为新手任务、七日任务、AI 生图、Prompt、邀请。
  • 登录后弹窗为单例配置，不展示统计卡、搜索筛选、新增弹窗、删除弹窗、草稿弹窗或弹窗启停列表逻辑。
  • 消息配置必须覆盖模板、业务落点、发送窗口、失败兜底和撤回原因。
  • 商城配置只管理商品展示、权益说明、积分消耗摘要和兑换异常，不进入支付、结算或卡密库存。
  • 数据看板必须标明指标口径、来源、排除规则、运营判断、负责人和下一步动作。
  • 批量动作必须限定范围，涉及状态、权限或展示变更时必须填写原因并留痕。
  • 详情抽屉必须包含对象摘要、状态历史、关联记录、风险原因和操作历史。
  • 异常态必须标明负责人、阻断原因和下一步处理动作。
  • 积分人工调整、权限变更和后台账号停用必须关联原因、复核状态和操作日志。
  • 危险操作必须二次确认并填写原因。
  • 日志字段必须包含对象、动作、前后状态、操作人、原因、时间。
  • 用户、Token、卡密、兑换凭证只能占位展示。
